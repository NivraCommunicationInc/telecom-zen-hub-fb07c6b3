## 1. Tokens / Sessions Security

### ✅ STATUS: IMPLEMENTED + VERIFIED

**Implementation:**
- All auth clients use `InMemoryStorage` (object JS in memory)
- `persistSession: false` configured on all clients
- `autoRefreshToken: false` configured on all clients

**Code Search Proof (grep results):**
```

src/integrations/supabase/portalClient.ts    → storage: inMemoryStorage, persistSession: false, autoRefreshToken: false

```

**Note:** `src/integrations/supabase/client.ts` uses localStorage (auto-generated by Lovable Cloud) but is used for public/anonymous operations only, not for authenticated sessions.

**E2E Test Coverage:** `e2e/security-token-check.spec.ts`
- ✅ Post page-load verification (localStorage, sessionStorage)
- ✅ Post-login form interaction verification
- ✅ IndexedDB check for token-like keys
- ✅ Cookie inspection for session artifacts
- ✅ Cross-navigation token leak prevention

**Verification Function:** `src/lib/inMemoryStorage.ts`
- `verifyNoStoredTokens()` - logs security status to console
- `purgeAllStoredTokens()` - cleans up on logout

**Trade-off:** Users must re-login after page refresh. This is the security cost for zero persistent token storage.

## 2. Extension in Public (pg_net)

### ⚠️ STATUS: ACCEPTED RISK (Cannot Mitigate - Proof Provided)

**Proof of REVOKE Attempt (2026-01-08):**
```sql
-- Attempted via migration:
REVOKE EXECUTE ON ALL FUNCTIONS IN SCHEMA net FROM PUBLIC;
REVOKE USAGE ON SCHEMA net FROM anon;
REVOKE USAGE ON SCHEMA net FROM authenticated;

-- Result: 
ERROR: 42501: must be owner of schema net
```

**Current Grants on net Schema Functions:**
| Function | Grantee | Privilege |
|----------|---------|-----------|
| _await_response | PUBLIC | EXECUTE |
| _encode_url_with_params_array | PUBLIC | EXECUTE |
| _http_collect_response | PUBLIC | EXECUTE |
| _urlencode_string | PUBLIC | EXECUTE |
| check_worker_is_up | PUBLIC | EXECUTE |
| http_collect_response | PUBLIC | EXECUTE |
| http_delete | PUBLIC | EXECUTE |
| http_get | PUBLIC | EXECUTE |
| http_post | PUBLIC | EXECUTE |
| wait_until_running | PUBLIC | EXECUTE |
| wake | PUBLIC | EXECUTE |
| worker_restart | PUBLIC | EXECUTE |

**Schema ACL (nspacl):**
```
{supabase_admin=UC/supabase_admin,=U/supabase_admin,supabase_functions_admin=U/supabase_admin,postgres=U/supabase_admin,anon=U/supabase_admin,authenticated=U/supabase_admin,service_role=U/supabase_admin}
```

**Why We Cannot Mitigate:**
- Schema `net` is owned by `supabase_admin`
- The `postgres` role (used by migrations) is not the owner
- REVOKE commands fail with `42501: must be owner of schema net`
- This is a Supabase/Lovable Cloud platform limitation, not a project configuration issue

**Compensating Controls:**
1. **No RLS policies reference net.*** - Verified by code search
2. **No client-side code calls net.*** - All HTTP calls use edge functions (service_role)
3. **Anon/authenticated cannot invoke net.* from browser** - Supabase SDK doesn't expose these functions

**Residual Risk:**
- If an attacker gains SQL injection access, they could potentially call `net.http_*` functions
- Mitigated by: parameterized queries throughout codebase, no raw SQL from user input

## 3. RLS Policies

### ✅ STATUS: VERIFIED - NO ALWAYS-TRUE POLICIES FOR ANON/AUTHENTICATED

**SQL Proof - "Always True" Policies:**
```sql
SELECT tablename, policyname, roles, qual, with_check
FROM pg_policies
WHERE schemaname = 'public'
  AND (qual::text = 'true' OR with_check::text = 'true');
```

**Result:** All "always true" policies are for `service_role` ONLY:
| Table | Policy | Roles | Purpose |
|-------|--------|-------|---------|
| admin_audit_log | Service role can insert audit logs | {service_role} | Backend audit logging |
| client_login_pins | Service role manages login pins | {service_role} | PIN auth system |
| email_queue | Service role inserts/updates email queue | {service_role} | Email processing |
| employee_operations_audit | Service role inserts operations audit | {service_role} | Audit trail |
| employee_pin_attempts | Service role manages PIN attempts | {service_role} | Security tracking |
| notifications | Service role inserts notifications | {service_role} | System notifications |
| rate_limit_attempts | Service role manages rate limits | {service_role} | Rate limiting |
| rate_limit_lockouts | Service role manages lockouts | {service_role} | Lockout management |
| rate_limits | Service role manages rate limits | {service_role} | Rate limiting |
| security_incidents | Service role inserts security incidents | {service_role} | Security logging |
| staff_otp_codes | Service role inserts OTP | {service_role} | 2FA system |

**SQL Proof - Anon SELECT Access (Non-Restrictive):**
```sql
SELECT tablename, policyname, qual
FROM pg_policies
WHERE schemaname = 'public'
  AND 'anon' = ANY(roles)
  AND cmd = 'SELECT'
  AND (qual IS NULL OR qual::text != 'false');
```

**Result:** Only 3 tables allow anon SELECT, all with proper restrictions:
| Table | Policy | Condition |
|-------|--------|-----------|
| site_offers | Anon read active offers | `is_active = true AND valid_from <= now() AND valid_until >= now()` |
| site_settings | Anon read public settings | `is_public = true` |
| streaming_catalog | Public view active catalog | `status = 'active'` |

**Access Tests:**
1. ✅ **Anon blocked from sensitive data**: All tables except the 3 above have `DENY anon USING(false)` policies
2. ✅ **User A cannot read User B**: All user-specific tables use `auth.uid() = user_id` in policies
3. ✅ **Public pages use only public views**: Verified by code review

**Tables with DENY for anon (28 total):**
- account_service_locations, accounts, activity_logs, appointments, authorized_users
- billing, channel_selections, client_activity_logs, client_documents, client_internal_notes
- client_streaming_subscriptions, contracts, employees, equipment_order_lines
- fulfillment_snapshots, ledger_entries, ledger_invoice_allocations, messages
- monthly_invoices, orders, payment_methods, profiles, replacement_internal_orders
- replacement_request_tickets, subscriptions, support_tickets, technicians, work_orders

## 5. Pre-Commit Checklist

### Database Changes
- [ ] Run `supabase--linter` before merge
- [ ] Verify no `USING (true)` for anon/authenticated on sensitive tables
- [ ] Add DENY policy for anon on any new sensitive table
- [ ] Never install extensions in `public` schema

### Frontend Changes
- [ ] No `localStorage.setItem` or `sessionStorage.setItem` for tokens
- [ ] Use only `inMemoryStorage` for auth clients
- [ ] Call `verifyNoStoredTokens()` in security-critical flows
- [ ] Public pages use only `*_public` views
