- ✅ IndexedDB - enumerate databases AND object stores for token patterns
- ✅ Cookies - check names AND values for token patterns

**CI Configuration Required:**
```yaml
env:
```

**Trade-off:** Users must re-login after page refresh. This is the security cost for zero persistent token storage.

## 2. Extension in Public (pg_net)

### ⚠️ STATUS: ACCEPTED RISK (Non-Exploitability Proven)

**Proof of REVOKE Attempt (2026-01-08):**
```sql
-- Attempted via migration:
REVOKE EXECUTE ON ALL FUNCTIONS IN SCHEMA net FROM PUBLIC;
REVOKE USAGE ON SCHEMA net FROM anon;
REVOKE USAGE ON SCHEMA net FROM authenticated;

-- Result: 
ERROR: 42501: must be owner of schema net
```

**Why We Cannot Mitigate:**
- Schema `net` is owned by `supabase_admin`
- The `postgres` role (used by migrations) is not the owner
- REVOKE commands fail with `42501: must be owner of schema net`

### Non-Exploitability Proofs

#### Proof A: Schema NOT exposed via PostgREST

```sql
SELECT current_setting('pgrst.db_schemas', true) as exposed_schemas;
-- Result: NULL (default = 'public' only)
```

The `net` schema is NOT in the exposed schemas list.

#### Proof B: RPC calls return 404 (NOT FOUND)

```bash
curl -X POST "https://xtgngmtxggascbxnswvb.supabase.co/rest/v1/rpc/http_get" \
  -H "apikey: [anon_key]" \
  -H "Content-Type: application/json" \
  -d '{"url": "https://example.com"}'
```

**Result:**
```json
{"code":"NOT_FOUND","message":"Requested function was not found"}
```

Even with `has_function_privilege('anon', oid, 'EXECUTE') = true`, the function is **NOT accessible** because the `net` schema is not exposed via PostgREST.

#### Proof C: No public wrapper functions

```sql
SELECT proname, prosrc 
FROM pg_proc p
JOIN pg_namespace n ON p.pronamespace = n.oid
WHERE n.nspname = 'public'
AND prosrc ILIKE '%net.%';
-- Result: 0 rows
```

**No functions in `public` schema call `net.*` functions.**

### Current Grants on net Schema Functions

| Function | anon_can_execute | auth_can_execute |
|----------|-----------------|------------------|
| http_get | true | true |
| http_post | true | true |
| http_delete | true | true |
| http_collect_response | true | true |
| (internal functions) | true | true |

**These grants are NOT exploitable** because the schema is not exposed via PostgREST.

### Residual Risk Assessment

| Risk Factor | Assessment |
|-------------|------------|
| Direct exploitation via PostgREST | **Impossible** (404 response proven) |
| Exploitation via wrapper function | **Impossible** (0 wrappers proven) |
| Exploitation via SQL injection | **Mitigated** (parameterized queries) |
| Overall residual risk | **Very Low** |

## 3. RLS Policies

### ✅ STATUS: VERIFIED - NO ALWAYS-TRUE POLICIES FOR ANON/AUTHENTICATED

**SQL Proof - "Always True" Policies:**
```sql
SELECT tablename, policyname, roles, qual, with_check
FROM pg_policies
WHERE schemaname = 'public'
  AND (qual::text = 'true' OR with_check::text = 'true')
  AND roles && ARRAY['anon', 'authenticated'];
```

**Result: 0 rows** - No "always true" policies for anon/authenticated on any table.

**All "always true" policies are for `service_role` ONLY** (backend audit logging, email queue, etc.)

**Access Tests:**
1. ✅ **Anon blocked from sensitive data**: All tables except public views have `DENY anon USING(false)` policies
2. ✅ **User A cannot read User B**: All user-specific tables use `auth.uid() = user_id` in policies
3. ✅ **Public pages use only public views**: Verified by code review

**Tables with DENY for anon (28 total):**
- account_service_locations, accounts, activity_logs, appointments, authorized_users
- billing, channel_selections, client_activity_logs, client_documents, client_internal_notes
- client_streaming_subscriptions, contracts, employees, equipment_order_lines
- fulfillment_snapshots, ledger_entries, ledger_invoice_allocations, messages
- monthly_invoices, orders, payment_methods, profiles, replacement_internal_orders
- replacement_request_tickets, subscriptions, support_tickets, technicians, work_orders

## 5. Pre-Commit Checklist

### Database Changes
- [ ] Run `supabase--linter` before merge
- [ ] Verify no `USING (true)` for anon/authenticated on sensitive tables
- [ ] Add DENY policy for anon on any new sensitive table
- [ ] Never install extensions in `public` schema

### Frontend Changes
- [ ] No `localStorage.setItem` or `sessionStorage.setItem` for tokens
- [ ] Use only `inMemoryStorage` for auth clients
- [ ] Call `verifyNoStoredTokens()` in security-critical flows
- [ ] Public pages use only `*_public` views
